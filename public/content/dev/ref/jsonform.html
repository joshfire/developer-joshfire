JSON form reference

The JSON form library is an open-source JavaScript library that can generate HTML forms from a structured data model and a layout description, validate user inputs against the declared data model, and create the resulting structured object initialized with user entered values upon form submission.

The library depends on the Underscore.js utility belt and on jQuery.

The library defines a JSON format to describe the underlying data model, possible default values, and the layout of the form. It re-uses existing concepts whenever possible.

Joshfire factory uses the JSON form format in the options section of the manifest file to allow templates to propose additional parameters and thus customize the functionality and look-and-feel of the generated Web application beyond datasources inputs.

JSON form format

Outline of a form declaration

The outline of a JSON form object is:

{
  "schema": Data schema,
  "value": Default values,
  "form": Form layout
}

Data schema

The schema property describes the structured data model that is to be generated. It is a JSON object that follows the JSON Schema specification. Using JSON Schema allows to define both the expected data structure and constraints about data values.

Below is a simple schema with two pieces of data, one that asks for a bit of HTML for an "about" page and another one that asks for a gender. The gender must be one of the enumerated choices:

"schema": {
  "type": "object",
  "properties": {
    "abouthtml": {
      "title": "About the application",
      "description": "HTML content that describes the application."
      "type": "string",
    },
    "gender": {
      "title": "Gender",
      "description": "Gender of the application."
      "type": "string",
      "enum": [
        "male",
        "female",
        "alien"
      ]
    }
  }
}

In just about all cases in JSON form, the root level of the JSON schema describes an object with properties. For the sake of simplicity and readability, you can drop this level and start directly with the contents of what its properties property would have been. The JSON form library will simply re-create the implicit root level if it cannot find the properties property. Using this shortcut notation, the previous example may be written as:

"schema": {
  "abouthtml": {
    "title": "About the application",
    "description": "HTML content that describes the application."
    "type": "string",
  },
  "gender": {
    "title": "Gender",
    "description": "Gender of the application."
    "type": "string",
    "enum": [
      "male",
      "female",
      "alien"
    ]
  }
}

Of course, do not use this shortcut notation if your schema needs to define a root-level property named "properties" otherwise the library would think you're actually using the complete notation!

Through recursion, you can build much more complex structures. The following schema would generate two levels:

"schema": {
  "abouthtml": {
    "title": "About the application",
    "type": "string"
  },
  "author": {
    "title": "Author",
    "type": "object",
    "properties": {
      "name": {
        "title": "Author's name",
        "type": "string"
      },
      "gender": {
        "title": "Author's gender",
        "type": "string",
        "enum": [
          "male",
          "female",
          "alien"
        ]
      },
      "magic": {
        "title": "Author's magic number",
        "type": "number",
        "minimum": 1
      }
    }
  }
}

Please refer to the JSON schema specification for more information. JSON form supports the following property types:

string
A string.
number
A number.
boolean
True or false.
array
An array of items. The items property must be defined for arrays.
object
For objects that have children. The properties property must be defined for objects.

If the type property and the items properties are not defined, string is assumed. If the type property is not defined but the items property is, object is assumed.

Default values

Use the value section to define default values for some of the data in the data model. The object's properties are the names of the properties defined in the schema section. To reference properties that are not at the root level, simply prefix the property name with the names of its ancestors, separating names with dots, e.g. author.gender in the example we've just seen.

The property values must be of the type defined in the schema.

Here are default values for our little example:

"value": {
  "abouthtml": "<p>This is a great app!</p>",
  "author.gender": "alien",
  "author.magic": 42
}

The JSON form library pre-fills a form input with its default value if one is defined. It also sets properties to their default values in the structured object generated upon form submission if the property was not rendered in the form.

Form layout

The form section defines the layout of the HTML form to generate. It is a JSON array. Each item in the array describes a section of the resulting form. The order of the sections in the generated HTML form follows the order of the items in the array.

An array item may be:

  1. A string with the specific value "*". The JSON form library generates a form section that consists of one form input per property defined in the schema. The default template is used for each form input. Use this specific value for simple forms. You will likely want more control over the layout as soon as your data model gets more complex, though.
  2. A string that references the name of a property in the schema (prefixed with ancestors names separated with dots for properties that are not at the root level). The JSON form library generates a form input for the property, using the default template.
  3. A JSON object that references a property in the schema. The JSON form library generates a form input for the property, using instructions in this JSON object to adjust the default template as necessary.
  4. A JSON object that describes a specific form section not linked to properties in the schema. Use this to add a submit button or a fieldset to group some of the form inputs in the generated form for example.

The following example generates a "default" form for all the properties defined in the data model. A "Submit" button is displayed at the end of the form. This submit button is one of the possible specific form sections that are not linked to properties in the schema.

"form": [
  "*",
  {
    "type": "submit",
    "value": "Submit"
  }
]

In the second example below, the abouthtml property expects an HTML fragment. The object tells the JSON form library to use a textarea element rather than the text input that the default template would generate for this field (note we get to defaults and ways to override defaults right after this example).

"form": [
  "author.name",
  "author.gender",
  {
    "key": "abouthtml",
    "type": "textarea"
  }
]

Default template

By default, the JSON form library checks the type and options of a property defined in the schema (string, boolean, enumeration defined...) to decide upon the type of HTML form input to generate (text, select, checkbox...). The following mapping is used:

  • If the property defines an enumeration in the JSON schema, a <select> element whose options are the enumeration values gets generated.
  • If the property type is string, a text input gets generated.
  • If the property type is number, a text input gets generated.
  • If the property type is boolean, a checkbox input gets generated.
  • If the property type is array, a fieldset gets generated. This fieldset contains a first input field whose type depends on the type defined for the array items in the JSON schema (through the items property) and a bit of HTML and JavaScript code to add/delete items.
  • If the property type is object, a fieldset gets generated. This fieldset contains the default form generated from the children of the object.

This mapping may evolve in the future, in particular to take advantage of the new form input types introduced in HTML5 when most browsers support them.

The JSON form library uses the information available in the JSON schema to complete the generated form section:

  • The title property serves as label for the input.
  • The description property is displayed next to the input field to guide user input.
  • The required property is used to assess the required nature of the input field. Required fields are advertized with an asterisk next to the field input. Required fields are checked upon form submission.

Form section description object

While it is tempting to stick to the use of strings that reference properties in the JSON schema, there are many reasons why you might want to get more control over generated form sections. We've already seen two examples: to generate a "submit" button or to display a larger text input when the expected user input may be long. There are other possible reasons such as:

  • It you ask for a password, you may want to generate a password input.
  • If your form gets more complex, you may want to introduce groups of form controls e.g. to ease readibility. You can do it but need to use a specific form section.
  • You may want to insert snippets of HTML among the other input fields to guide users.
  • You may want to provide your own validation error messages.
  • You may simply want to replace part or all of the HTML markup that would normally get generated by your own.

You can do all of the above through properties of the JSON object that you can provide as item of the form's array. Possible properties are defined below. The list is sorted by alphabetic order. Most of the time, you will set the key property (when you want to reference a property in the JSON schema) and/or the type property. The additional properties that you may set on top of key and type are usually a direct consequence of the values you set for them.

append
Use this property to insert a suffix after the generated form input.
The property accepts an HTML fragment string as value.
Note the JSON form library does not append the suffix to the generated object upon form submission.
(see also prepend)
id
By default, the generated form input has an id of the formjsonform-elt-[key]. Set this property to override the id attributed of the generated form input.
items
This property is only useful when the type property is set to fieldset or actions.
This property accepts an array of form section description objects as value. Use this property to introduce sub-levels and grouping of form inputs in the generated form. See examples below for details.
itemshtml
This property is only useful when the type property is set to fieldset or actions.
Use this property to take entire control over the generated HTML content of the <fieldset> element or <div> element.
The items property must not be defined when this property is defined.
This property accepts an HTML fragment string as value.
key
The name of the property in the JSON schema bound to this form section.
This property must be set to link this form section to a property in the JSON schema. Conversely, this property must not be set to define a specific form section (such as a "submit" button).
Property names must be prefixed with their ancestors names separated with a dot if they are not first level properties (e.g. "author.name" in the recurring example of this documentation).
legend
This property is only useful when the type property is set to fieldset.
Set this property to define the legend for the generated fieldset.
name
By default, the name attribute of the generated form input is set to the name of the property in the JSON schema. Set this property to override it.
prepend
Use this property to insert a prefix before the generated form input.
The property accepts an HTML fragment string as value.
Note the JSON form library does not insert the prefix to the generated object upon form submission.
(see also append)
template
This property is only useful when the type property is not set to fieldset or actions.
Use this property to take entire control over the generated HTML form input (label and actual form input included).
This property accepts an HTML fragment string as value.
title
By default, the label of the generated form input is set to the title defined for the property in the JSON schema. Set this property to override the title (or to define it if the schema does not).
type
The type of form section to generate.
This property must be set to define a specific form section (such as a "submit" button). It may be set when this form section is bound to a property in the JSON schema to override the type of form input that would be generated by default.
This property accepts the following values:
  • one of the type attribute values defined in HTML5, although please note that the JSON form library currently only supports a fraction of these values, namely text, password, checkbox, file, submit and button. Support for additional values will be added over time. The generated form input gets adapted to use the appropriate input type as a consequence.
    The key property must not be defined when the value is button or submit.
  • textarea to force the use of a <textarea> element for the form input.
  • fieldset to generate a <fieldset> element. The key property must not be defined when this value is used.
  • actions to generate a section meant to contain possible user actions. The key property must not be defined when this value is used.
value
The JSON form library pre-fills the generated form input with the value of this property unless one is already defined in the value section. In practice, this property is typically useful to set the label of a submit button.

The following declaration tells the JSON form library to generate a form that consists of two groups of form inputs: the first group for inputs related to the application and the second group for inputs related to the author of the application. A submit button, encapsulated in an "actions" section closes the form.

"form": [
  {
    "type": "fieldset",
    "legend": "Information about the application",
    "items": [
      {
        "key": "abouthtml",
        "type": "textarea"
      }
    ]
  },
  {
    "type": "fieldset",
    "legend": "Information about the author of the application",
    "items": [
      {
        "key": "author.name",
        "title": "Nickname",
        "prepend": "Master"
      },
      "author.gender",
      {
        "key": "author.magic",
        "type": "password",
      }
    ]
  },
  {
    "type": "actions",
    "items": [
      {
        "type": "submit",
        "value": "Submit"
      }
    ]
  }
]

How to style the generated form

Most of the markup generated by the JSON form library is composed of raw HTML5 tags. When more semantic is needded, the library follows the class names of the Bootstrap toolkit, e.g. clearfix, input, success, warning, error, help-inline, input-prepend, etc.

Using the JSON form library

Joshfire factory users may skip this section: the factory already integrates the JSON form library and exposes the appropriate structured data object to templates.

Using the library in a Web page

The library is a simple JavaScript file that you need to include in the list of scripts of your Web application. It requires the following global objects to be available:

  • $ defined in jQuery
  • _ defined in Underscore.js
  • JSON which should be defined in all modern Web browsers (but you may need to use a dedicated JavaScript library to support older Web browsers)

The library adds two jQuery functions:

jsonForm
The function takes a JSON object that follows the JSON form format as argument. It generates the corresponding HTML form which is used as HTML content of the DOM element the function is applied to. 
$("#form").jsonForm(JSON form format object);
jsonFormValue
The function does not take any argument. It returns the current structured values object of the JSON form attached to the DOM element the function is applied to, following the data model defined with the form. 
$("#form").jsonForm(JSON form format object);
// Let the user fill out the form
[...]
var dataObject = $("#form").jsonFormValue();
